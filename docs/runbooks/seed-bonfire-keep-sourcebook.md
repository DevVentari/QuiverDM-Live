# Runbook: Seed "Tales from The Bonfire Keep" as a Master Sourcebook

Imports the DM's homebrew world **Hameria Ire** (campaign: *Tales from The Bonfire
Keep*) as a canonical **master sourcebook**, stored exactly like the published books
(Curse of Strahd, Icewind Dale, Lost Mine). This is the "a DM brings their existing
prep into QuiverDM" test path.

## What gets created

- **`DdbSourcebook`** `slug=tales-bonfire-keep`, title *"Tales from The Bonfire Keep"*,
  owned by the QuiverDM library user `cmpl6071u0001srcmhadoz41h` (the same owner as
  `cos` / `lmop` / `idrotf` / `wdh` / `toa` / `bgdia`).
- **28 `DdbSourcebookChapter`** rows (377 `bodySections`), loaded from the DM's Hugo
  site content — world lore, factions, NPCs, bestiary, mechanics, and adventures.
- **`SourcebookEntity`** rows:
  - Hand-authored **FACTION / THREAT / SECRET** (12) — types the generic extractor
    cannot emit but which this faction-centric world needs.
  - Auto-extracted **NPC / LOCATION / ITEM / EVENT** via the dual-model pipeline.

## Source of truth

The DM's Hugo export (`HugoSite/content/**`) is committed under
`scripts/fixtures/bonfire-keep/content/`. The stale `final_json/` from the export is
**deliberately ignored** — entities are derived fresh from the source markdown through
the current extraction pipeline.

> Two Hugo files (`mechanics/systems.md`, `mechanics/items.md`) were placeholder stubs
> in the export; the real content was swapped in from the Obsidian vault
> (`Tales From The Bonfire Keep/Systems.md`, `Items.md`).

## Re-run

Two idempotent steps. Step 1 is fully self-contained; step 2 replays the committed
agent-extracted JSON (no AI calls needed).

```bash
# 1. Chapters (bodySections) + hand-authored FACTION/THREAT/SECRET entities.
npx tsx scripts/seed-bonfire-keep-sourcebook.ts            # add --dry-run to preview

# 2. Upsert the NPC/LOCATION/ITEM/EVENT entities from the committed extraction JSON.
npx tsx scripts/upsert-bonfire-entities.ts
```

### How the entity JSON was produced

`scripts/fixtures/bonfire-keep/extracted/*.json` was generated by **subagents** (one per
chapter-group) reading the chapter markdown and emitting the extraction schema
(`{ chapterSlug, npcs, locations, items, encounters }`). This was chosen over the external
AI pipeline because all paid/free LLM providers were rate-limited or out of credit at the
time, and because committing the JSON makes the seed reproducible without any AI dependency.

To re-extract from scratch (e.g. after editing the source markdown), re-dispatch the
extraction subagents to regenerate the JSON, then re-run step 2.

### Alternative: the generic extraction pipeline

`create-master-sourcebook.ts` can also auto-extract over the chapter `bodySections`. It now
accepts `--providers <a,b,...>` (keys: `claude`, `openai`, `gemini`, `groq`, `ollama`;
default `claude,openai`), and `src/lib/ai/extract-chapter-entities.ts` retries on
rate-limit (429) errors honouring each provider's `retry-after`:

```bash
npx tsx scripts/create-master-sourcebook.ts --slug tales-bonfire-keep \
  --skip-crawl --skip-images --providers claude,openai
```

All paths upsert on `(sourcebookId, type, name)`, so they compose safely. Note: a mixed run
can leave near-duplicate names across extractors (e.g. `Ambric` vs `Ambric — Anchor of
Justice`); prefer a single extraction source per run.

## Link into a campaign (the actual import test)

In the app: **Sourcebooks → Tales from The Bonfire Keep → Add to campaign**, or call
`ddbSync.linkSourcebookToCampaign({ campaignId, sourcebookId })`. That triggers
`seedCampaignFromSourcebook` (see `src/server/repositories/ddb-sync.repository.ts`),
which clones the world's homebrew + WorldEntities into the campaign.
